# AGENTS.md

## Project

`chess-crawl` is a local-first Python CLI for building a chess data archive.

The project is intended to retrieve public chess data from providers such as Chess.com and Lichess, preserve retrieved raw data locally, and normalize selected records into SQLite for querying, reporting, and bounded discovery.

Core principles:

* Provider-neutral architecture.
* Raw-first local storage.
* SQLite-backed persistence.
* Bounded retrieval.
* Public APIs only.
* No scraping.
* No rate-limit evasion.
* No cheating accusations or automated misconduct labels.
* No cross-provider identity inference.

A username on one chess provider must not be assumed to identify the same human on another provider.

## Agent Permissions

Agents may:

* Inspect the repository.
* Edit files.
* Run local commands and tests.
* Create local git commits.

Agents may not:

* Push to any remote.
* Open pull requests.
* Merge branches.
* Change remotes.
* Change Git configuration.
* Rewrite published history.
* Add credentials, tokens, or secrets.
* Run large or unbounded live API crawls unless explicitly instructed.

## Git Rules

Before making changes, run:

```bash
pwd
git status --short --branch
```

Before finishing, run:

```bash
git status --short
```

Local commits are allowed when the work is complete and tested.

Use concise, project-focused commit messages:

```text
chore: initialize archive schema
feat: add provider registry
test: cover raw payload idempotency
docs: add bounded crawl notes
```

Do not push.

Do not open pull requests.

## Attribution Rules

Contribution and attribution must not include agents, models, tools, or automation systems.

Do not include agent/tool names in:

* Commit authorship.
* Commit messages.
* Commit trailers.
* `Co-authored-by` trailers.
* `Signed-off-by` trailers.
* README acknowledgements.
* CHANGELOG entries.
* AUTHORS or contributors files.
* Release notes.
* Public documentation.

Do not write attribution such as “generated by,” “implemented by,” or “co-authored by” an agent, model, or tool.

Reports and handoffs should describe the work performed, not who or what performed it.

## Working Style

Prefer small, coherent changes.

Do not jump ahead of the requested task.

Use the current user request as the active instruction. When a task references a plan, report, design note, or handoff file, read that file for the task. Do not assume temporary planning files are permanent project authority.

Keep implementation aligned with these invariants:

* Preserve raw retrieved data before or alongside normalization.
* Keep users and games provider-scoped.
* Keep retrieval bounded by default.
* Keep default tests offline.
* Keep provider-specific API behavior isolated in provider-specific modules.
* Keep shared storage and reports provider-neutral.

## Testing

Run the fastest relevant tests before committing.

Default validation:

```bash
python -m pytest
```

When CLI code changes, also check CLI help using the implemented entrypoint, for example:

```bash
python -m chess_crawl --help
```

or:

```bash
python -m chess_crawl.cli --help
```

Do not add default tests that require internet access.

Live API tests, if added, must be explicitly marked and skipped by default.

Do not make tests sleep in real time for rate-limit behavior. Use fake clocks, monkeypatching, or fixtures.

## Data and Secrets

Do not commit:

* SQLite databases.
* Raw provider dumps.
* `.env` files.
* API tokens.
* OAuth tokens.
* Local caches.
* Generated exports.
* Temporary smoke-test artifacts.

Use `.env.example` only for placeholder, non-secret configuration.

## Provider and Safety Boundaries

Use documented public APIs only.

Do not scrape HTML pages.

Do not use browser automation.

Do not evade rate limits.

Do not parallelize crawling unless explicitly approved.

Do not add features that label, score, rank, or accuse players of cheating.

Provider account-status fields may be stored only as provider-supplied neutral facts.

Reports must not imply guilt or misconduct.

## Coordination Notes

Use project-local coordination files only when the current task calls for them.

Handoff notes should include:

* Objective.
* Summary of changes.
* Files changed.
* Commands added or changed.
* Tests added.
* Tests run and results.
* Known limitations.
* Deviations from the active task or referenced plan.
* Recommended next work.
* Risks or decisions to preserve.

Coordination notes must not include agent attribution.
