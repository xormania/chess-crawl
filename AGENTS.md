# AGENTS.md

## Project

`chess-crawl` is a local-first Python CLI for building a provider-neutral chess data archive.

The project is intended to retrieve public chess data from providers such as Chess.com and Lichess, preserve the raw provider responses locally, and normalize selected data into SQLite tables for querying, reporting, and bounded graph discovery.

Core principles:

* Provider-neutral design.
* Raw-first local archive.
* SQLite-backed storage.
* Bounded retrieval only.
* Public APIs only.
* No scraping.
* No rate-limit evasion.
* No cheating accusations or automated misconduct labels.
* No cross-provider identity inference.

A Chess.com username and a Lichess username must never be assumed to refer to the same human.

* Open pull requests.

* Change remotes.
* Change GitHub settings.

Local commits are allowed.

## Git Rules

Before making changes, run:

```bash
pwd
git status --short --branch
```

After making changes, run:

```bash
git status --short
```

Agents may create local commits when the work is complete and tested.

Commit messages should be concise and project-focused, for example:

```text
chore: initialize archive schema
feat: add provider registry
test: cover raw payload idempotency
docs: add bounded crawl notes
```

Do not push commits.

Do not open PRs.

Do not add agent attribution to Git history.

## Working Style

Prefer small, coherent changes.

Do not jump ahead of the requested phase or task.

Read the relevant plan or handoff first:

```bash
cat proj/plan.md
```

When working across phases, also read the latest phase report:

```bash
ls proj/
```

Keep implementation aligned with the project plan, but do not preserve old wording blindly if the plan has been superseded by explicit user instruction.

When unsure, preserve the project’s core invariants:

* Raw provider payloads must be preserved before or alongside normalization.
* Users and games are provider-scoped.
* Retrieval must be bounded.
* Default tests must be offline.
* Provider-specific behavior belongs behind provider-specific modules.
* Shared storage and reports must remain provider-neutral.

## Testing

Run the fastest relevant tests before committing.

Default validation:

```bash
python -m pytest
```

Also check CLI help when CLI code changes:

```bash
python -m chess_crawl --help
```

or, depending on the implemented entrypoint:

```bash
python -m chess_crawl.cli --help
```

Do not add default tests that require internet access.

Live API tests, if ever added, must be explicitly marked and skipped by default.

Do not make tests sleep in real time for rate-limit behavior. Use fake clocks, monkeypatching, or fixtures.

## Data and Secrets

Do not commit:

* SQLite databases.
* Raw provider dumps.
* `.env` files.
* API tokens.
* OAuth tokens.
* Local caches.
* Generated exports.
* Temporary smoke-test artifacts.

Expected ignored patterns include:

```text
*.sqlite
*.sqlite3
*.db
.env
.env.*
data/
tmp/
.cache/
```

Use `.env.example` only for non-secret placeholder configuration.

## Provider and Safety Boundaries

Use documented public APIs only.

Do not scrape HTML pages.

Do not use browser automation.

Do not evade rate limits.

Do not parallelize crawling unless explicitly approved.

Do not add features that label, score, rank, or accuse players of cheating.

Provider account-status fields may be stored only as provider-supplied neutral facts.

Any report touching account status or anomaly-like statistics must avoid conclusions and must not imply guilt.
